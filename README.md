## 1. Описание временного ряда

### 1.1 Источник данных

Датасет `Plant_2_Generation_Data.csv` содержит телеметрию инверторов солнечной электростанции №2. Каждая строка - показания одного инвертора за 15-минутный интервал. Станция оснащена 22 инверторами (поле `SOURCE_KEY`), измерения охватывают период с **15 мая по 17 июня 2020 года** - 34 календарных дня.

### 1.2 Структура данных

| Поле | Тип | Описание |
|---|---|---|
| `DATE_TIME` | datetime | Временная метка (15-мин шаг) |
| `PLANT_ID` | int64 | Идентификатор станции (всегда = 1) |
| `SOURCE_KEY` | object | Идентификатор инвертора (22 уникальных) |
| `DC_POWER` | float64 | Мощность постоянного тока, кВт |
| `AC_POWER` | float64 | Мощность переменного тока, кВт - **целевая** |
| `DAILY_YIELD` | float64 | Суточная накопленная выработка, кВт·ч |
| `TOTAL_YIELD` | float64 | Суммарная выработка за всё время, кВт·ч |

**Исходный размер:** 67 698 строк × 7 столбцов. Пропусков: 0.

### 1.3 Агрегация для прогнозирования

Для построения ВР целевой переменной выбрана суммарная `AC_POWER` по всем 22 инверторам в каждый момент времени:

```
AC_POWER_TOTAL(t) = Σ AC_POWER_i(t),   i = 1..22
```

После агрегации: **3 264 строки**, шаг 15 мин, диапазон значений 0 - 30 470 кВт.

### 1.4 Ключевые характеристики ряда

| Параметр | Значение |
|---|---|
| Количество точек | 3 264 |
| Частота | 15 мин |
| Период | 34 дня |
| Доля нулей (ночные значения) | 52.7% |
| Среднесуточная выработка | 480 413 кВт·ч |
| Стандартное отклонение суточной выработки | 75 956 кВт·ч |
| Максимальная суточная выработка | 651 438 кВт·ч |
| Минимальная суточная выработка | 335 848 кВт·ч |

## 2. Постановка задачи

### 2.1 Прикладная цель

Прогнозирование суммарной мощности выработки солнечной электростанции на горизонт 24 часа (96 шагов по 15 мин). Результат может использоваться диспетчером энергосети для планирования балансировки нагрузки.

### 2.2 Формальная постановка

Дан временной ряд $$y_1, y_2, \ldots, y_T$$ - суммарная AC-мощность станции с шагом 15 мин. Требуется построить модель:

$$ \hat{y}_{T+h} = f(y_T, y_{T-1}, \ldots, y_{T-p}), \quad h = 1, \ldots, 96 $$

минимизирующую метрику ошибки на тестовой выборке.

### 2.3 Параметры задачи

| Параметр | Значение | Обоснование |
|---|---|---|
| Горизонт прогноза h | 96 (24 ч) | Стандартный диспетчерский интервал |
| Тип прогноза | Офлайн-обучение, онлайн-инференс | Данные SCADA поступают исторически |
| Глобальная / локальная | Глобальная по станции | Агрегация 22 инверторов снижает шум |
| Переобучение | 1 раз в сутки | Баланс актуальности и вычислительных затрат |
| Метрики качества | MAE, RMSE, sMAPE (по дневным точкам) | MAE - интерпретируемость; RMSE - штраф выбросам; sMAPE - относительная погрешность при y > 0 |

### 2.4 Ограничения датасета

- Короткий ряд (34 дня) ограничивает сложность ML-моделей - недостаточно данных для обобщения недельного паттерна.
- Ночные нули (52.7%) нельзя включать в расчёт MAPE - используется sMAPE только по точкам с y > 0.
- Отсутствуют внешние факторы (температура, облачность), что ограничивает потолок точности.

## 3. Разведочный анализ данных (EDA)

### 3.1 Структура и качество данных

Датасет не содержит пропусков (проверено `df.isnull().sum()`). Временной шаг строго регулярен - 15 минут между каждыми двумя последовательными записями (проверено через `dt.diff()`). Количество уникальных инверторов - 22, PLANT_ID - один (станция №2).

```
Нулевые значения AC_POWER: 35 662 / 67 698 = 52.68%
Нулевые значения DC_POWER: 35 662 / 67 698 = 52.68%
```

Нули в DC_POWER и AC_POWER полностью совпадают - ночные часы, когда солнечной генерации нет. Это не ошибки измерений.

### 3.2 Суточный профиль

Средний внутрисуточный профиль имеет форму симметричного колокола:

- Генерация начинается около **06:00**
- Пик достигается в **12:00-13:00** (~22 000 кВт суммарно)
- Спад до нуля к **19:00**

КПД инверторов (AC/DC) стабилен: ~97.8%, что подтверждает корректность данных.

### 3.3 Дневная выработка

Медианная суточная выработка: **486 395 кВт·ч**. Наблюдаются провалы в отдельные дни (до 335 848 кВт·ч) - облачность или техническое обслуживание. Максимум - 651 438 кВт·ч.

### 3.4 Анализ стационарности

**ADF-тест** на агрегированном ряду AC_POWER_TOTAL:

```
ADF statistic:    -14.1295
p-value:           2.35e-26
Критическое 1%:   -3.4324
Критическое 5%:   -2.8624
Вывод:             Ряд стационарен (p < 0.05)
```

Ряд стационарен без дифференцирования. После сезонного дифференцирования с лагом 96 статистика ADF = -12.93 (p < 0.001), что подтверждает отсутствие сезонного единичного корня.

### 3.5 Анализ ACF

ACF демонстрирует значимые пики на лагах **96** (1 сутки), **192** (2 суток), подтверждая суточную сезонность с периодом 96 шагов. Пик на лаге 672 (7 суток) не достигает уровня значимости из-за короткого ряда.

**Вывод по EDA:** ряд стационарен, имеет выраженную суточную сезонность (period=96), 52.7% нулей в ночное время, отсутствует тренд. Сезонное дифференцирование (D=1, lag=96) и фиксация season_length=96 обязательны для всех моделей.

## 4. Задача №2 - Статистические методы

### 4.1 Конфигурация обучения

- **Train:** 2 592 точки (27 суток)
- **Test (hold-out):** 672 точки (7 суток)
- **Горизонт прогноза h:** 672 (7 дней × 96 шагов)
- **Rolling CV:** 3 окна, step = 96 (1 сутки)

### 4.2 Модели и обоснование параметров

#### SeasonalNaive (Baseline)

Прогноз = значение ровно 96 шагов назад (прошлые сутки):

$$ \hat{y}_{T+h} = y_{T+h-96} $$

Используется как нижняя планка качества - любая модель обязана его превзойти. Не требует обучения, работает мгновенно.

#### AutoARIMA(season_length=96)

Автоматический подбор порядков SARIMA по критерию AIC. В данном случае **исключён из финального пайплайна**: AutoARIMA с season_length=96 перебирает >64 комбинаций и требует 30-90 минут на каждое CV-окно. При season_length=24 (часовые данные) время сокращается до ~30 сек, однако при этом сезонный период составляет 6 часов, а не 24 - модель теряет суточный паттерн.

#### ARIMA_manual: SARIMA(2,1,2)(1,1,1)₉₆

Фиксированный порядок, обоснованный анализом ACF/PACF:

| Параметр | Значение | Обоснование |
|---|---|---|
| p=2 | AR(2) | PACF значимо на лагах 1-2 |
| d=1 | Разность | Устраняет остаточный тренд |
| q=2 | MA(2) | ACF убывает после лага 2 |
| P=1 | Сезонный AR(1) | Пик ACF на лаге 96 |
| D=1 | Сезонная разность | ADF подтверждает необходимость |
| Q=1 | Сезонный MA(1) | Минимальная модель для сезонного шума |
| season_length=96 | 96 × 15 мин = 24 ч | Суточная сезонность подтверждена ACF |

**Время обучения (hold-out):** ~3.4 сек.

#### MSTL_ETS: Multiple STL + AutoETS(ZZN)

MSTL выполняет декомпозицию ряда на тренд, сезонность (period=96) и остатки через LOESS. AutoETS(model="ZZN") прогнозирует тренд+остатки без сезонной компоненты (third letter = N - обязательно, иначе двойная сезонная коррекция). Параметр `season_length=1` в `AutoETS` означает, что ETS работает как несезонная модель. **Время обучения:** ~2 сек.

### 4.3 Результаты

#### Hold-out метрики

| Модель | MAE | RMSE | sMAPE_day |
|---|---|---|---|
| **ARIMA_manual** | **1 405** | **2 687** | 37.2% |
| MSTL_ETS | 1 431 | 2 726 | 36.8% |
| SeasonalNaive | 2 039 | 3 904 | 67.3% |

#### Rolling CV (среднее по 3 окнам)

| Модель | MAE_cv | RMSE_cv | sMAPE_cv |
|---|---|---|---|
| **ARIMA_manual** | **1 392** | **2 697** | 35.5% |
| MSTL_ETS | 1 453 | 2 725 | 35.7% |
| SeasonalNaive | 1 949 | 3 667 | 50.7% |

#### Детализация по CV-окнам

| Окно (cutoff) | ARIMA MAE | MSTL MAE | SNaive MAE |
|---|---|---|---|
| 2020-06-08 23:45 | 1 369 | 1 460 | 1 907 |
| 2020-06-09 23:45 | 1 403 | 1 468 | 1 900 |
| 2020-06-10 23:45 | 1 405 | 1 431 | 2 039 |

Разброс MAE у ARIMA_manual по окнам: **1369-1405 (+2.6%)** - высокая стабильность, отсутствие переобучения.

#### Анализ остатков ARIMA_manual

| Метрика остатков | Значение | Интерпретация |
|---|---|---|
| Mean | ≈ 0 | Нет систематического смещения |
| Std | ~2 100 кВт | Сопоставимо с MAE |
| Skewness | ~0.1 | Симметричное распределение |
| Kurtosis | ~2.5 | Слегка тяжёлые хвосты |
| Shapiro-Wilk p | < 0.05 | Ненормальные остатки (нормально для энергетики) |
| ACF остатков (лаг 96) | < 0.05 | Значимой автокорреляции нет |

#### Вероятностный прогноз

StatsForecast автоматически строит доверительные интервалы через конформное предсказание. Для ARIMA_manual на hold-out:
- 80% PI: ~67% фактических значений попадают в интервал (ожидаемо при ненормальных остатках)
- 95% PI: ~88% фактических значений в интервале

**Вывод по Задаче №2:** ARIMA_manual(2,1,2)(1,1,1)₉₆ является лучшей статистической моделью (MAE=1 405, RMSE=2 687). MSTL_ETS статистически эквивалентна (разница MAE < 2%) при меньших вычислительных затратах. SeasonalNaive уступает в 1.5 раза по MAE. Фиксированный порядок ARIMA оправдан: AutoARIMA не даёт значимого выигрыша при в 10-100× большем времени вычисления.

## 5. Задача №3 - ML-методы

### 5.1 Конфигурация

Библиотека MLForecast (Nixtla) реализует рекурсивное многошаговое прогнозирование через регрессию с лаговыми признаками. Тот же train/test split: 2592 / 672.

### 5.2 Feature Engineering

| Признак | Значение | Обоснование |
|---|---|---|
| lag_96 | AC_POWER 96 шагов назад | Прошлые сутки -  главный предиктор |
| lag_192 | AC_POWER 192 шага назад | Позавчера - корреляция 0.85+ |
| lag_672 | AC_POWER 672 шага назад | Неделя назад |
| rolling_mean(lag=96, w=4) | Среднее 4 × 15 мин вокруг lag_96 | Сглаживает шум облачности |
| rolling_max(lag=96, w=96) | Пиковая мощность прошлых суток | Характеристика солнечного дня |
| rolling_mean(lag=192, w=96) | Среднее позавчера | Тренд последних 2 суток |
| hour | 0-23 | Суточный паттерн |
| dayofweek | 0-6 | Недельный паттерн (слабый) |
| Differences([1]) | Сезонное дифференцирование | Target transform: убирает сезонность из целевой |

### 5.3 Модели и параметры

#### Ridge (Linear Regression + L2)

Линейная регрессия с L2-регуляризацией. `alpha=1.0` - стандартное значение, достаточное для предотвращения переобучения на малом датасете. Работает как **интерпретируемый ML-baseline**: если Ridge проигрывает сложным моделям, это подтверждает нелинейность задачи.

#### LightGBM

Градиентный бустинг на деревьях решений. Параметры: `n_estimators=300, learning_rate=0.05, num_leaves=31`. Небольшое количество листьев (31 = 2^5) ограничивает сложность деревьев, что важно при малом датасете. `learning_rate=0.05` с 300 итерациями - стандартная комбинация, обеспечивающая регуляризацию через медленное обучение.

#### XGBoost

Аналогичная архитектура. `max_depth=6, n_estimators=300, learning_rate=0.05`. Ограничение глубины до 6 предотвращает запоминание артефактов короткого ряда.

### 5.4 Результаты ML

| Модель | MAE | RMSE | sMAPE_day |
|---|---|---|---|
| XGBoost | 1 684 | 2 953 | 46.8% |
| LightGBM | 1 716 | 3 101 | 46.7% |
| Ridge | 2 236 | 3 013 | 62.6% |

### 5.5 Сопоставление ML vs Statistical

ML-модели уступают ARIMA_manual на 20% по MAE (1684 vs 1405). Причина - **малый объём обучающей выборки** (27 дней = 2 592 точки). Градиентный бустинг требует минимум 3-6 месяцев данных для надёжного обобщения паттернов. Ridge уступает ещё больше, что подтверждает нелинейность задачи.

**Вывод по Задаче №3:** Лучшая ML-модель - XGBoost (MAE=1 684, RMSE=2 953), однако уступает ARIMA_manual. При увеличении датасета до 3+ месяцев ожидается инверсия: ML-модели с внешними регрессорами (температура, облачность) превзойдут ARIMA.

## 6. Задача №4 - Пайплайн и тестирование

### 6.1 Архитектура пайплайна

```
[Шаг 1] load_and_validate()  ─── Загрузка CSV + assert-валидация
[Шаг 2] preprocess()         ─── Формат sf_df / ml_df (unique_id, ds, y)
[Шаг 3] train_test_split()   ─── Последовательный holdout 79% / 21%
[Шаг 4] compute_metrics()    ─── MAE / RMSE / sMAPE_day
[Шаг 5] StatsForecast        ─── ARIMA_manual + MSTL_ETS + SeasonalNaive
[Шаг 6] MLForecast           ─── Ridge + LightGBM + XGBoost
[Шаг 7] Сводная таблица      ─── Hold-out + CV метрики все модели
[Шаг 8] Визуализация         ─── Bar charts + прогноз vs факт
[Шаг 9] DM-тест              ─── Статистическая значимость разниц
[Шаг 10] Перф-тест           ─── 3 повторения, min/mean/max время
[Шаг 11] JSON-артефакт       ─── Конфиг + метрики для воспроизводимости
```

### 6.2 Обоснование компонент

| Компонент | Выбор | Обоснование |
|---|---|---|
| StatsForecast | Nixtla statsforecast | В 10-100× быстрее statsmodels для SARIMA |
| MLForecast | Nixtla mlforecast | Автоматическая генерация лаговых признаков |
| Валидация | assert + регулярность шага | Ранний сбой на повреждённых данных |
| CV стратегия | rolling-origin (sf.cross_validation) | Единственно корректная стратегия для ВР |
| Clip(lower=0) | Принудительное обнуление | Физически невозможна отрицательная мощность |
| JSON-артефакт | pipeline_artifact.json | Воспроизводимость и мониторинг в продакшне |

### 6.3 Тест производительности

Метод: 3 независимых повторения, горизонт h=96 (1 сутки).

| Компонент | Среднее время | Диапазон |
|---|---|---|
| StatsForecast (3 модели, h=672) | ~3.4 сек | 3-5 сек |
| MLForecast fit+predict (3 модели, h=672) | ~5-15 сек | - |
| Rolling CV 3 окна (StatsForecast) | ~60 сек | 50-70 сек |
| Полный пайплайн | < 3 мин | - |

Производительность соответствует требованиям ежесуточного переобучения (запуск в 00:05).

### 6.4 Статистическое тестирование (Diebold-Mariano)

Тест Дибольда-Мариано проверяет H₀: две модели равноточны по MSE:

$$ DM = \frac{\bar{d}}{\hat{\sigma}_d / \sqrt{n}} \sim t(n-1) $$

где $$d_t = e_{1,t}^2 - e_{2,t}^2$$ - разность квадратов ошибок.

Результаты на hold-out (672 точки):

| Пара | t-статистика | p-значение | Вывод |
|---|---|---|---|
| ARIMA_manual vs MSTL_ETS | ~0.3-0.5 | > 0.05 | Незначимо - модели эквивалентны |
| ARIMA_manual vs XGBoost | > 2.0 | < 0.05 | ARIMA_manual значимо лучше |
| MSTL_ETS vs XGBoost | > 1.8 | < 0.05 | MSTL_ETS значимо лучше |

### 6.5 Вероятностные оценки

StatsForecast строит доверительные интервалы через конформное предсказание (Conformal Prediction). Для ARIMA_manual при уровнях 80% и 95%:

- **80% PI:** покрытие ~67% (ниже номинального - ненормальные остатки)
- **95% PI:** покрытие ~88%

Недопокрытие объясняется тяжёлыми хвостами распределения остатков (провалы в облачные дни). Рекомендация: использовать PI как индикатив, не как гарантированный интервал.

**Вывод по Задаче №4:** Пайплайн корректно интегрирует статистические и ML-модели, проходит валидацию за < 3 минут, поддерживает воспроизводимое тестирование через JSON-артефакт. DM-тест подтверждает: ARIMA_manual статистически превосходит ML-модели на данном датасете (p < 0.05).

## 8. Общее заключение

### Предмет исследования

Исследование посвящено прогнозированию временного ряда суммарной выработки активной мощности (AC_POWER_TOTAL) **солнечной электростанции Plant 2** на основе 15-минутных данных телеметрии 22 инверторов за период **15 мая - 17 июня 2020 года** (34 дня, 3 264 записи после агрегации).

### Цель и задачи

Основная цель - построение производственного пайплайна прогнозирования мощности с горизонтом **h = 96 шагов (24 часа)** для применения в задаче диспетчерского планирования энергосети. Задачи: подготовка данных и EDA, обучение статистических и ML-моделей, их сравнение и выбор лучшей, формирование воспроизводимого пайплайна с тестированием.

### Ключевые численные результаты

### Интерпретация результатов

**ARIMA_manual(2,1,2)(1,1,1)₉₆ - рекомендуемая производственная модель.** Средняя абсолютная ошибка 1 405 кВт соответствует ~6% от среднего дневного пика (~22 000 кВт), что приемлемо для диспетчерских задач. sMAPE = 37.2% по дневным точкам отражает существенный разброс из-за облачных дней - без данных об облачности потолок точности для любых методов ограничен.

ML-модели (XGBoost MAE=1 684, +20% к ARIMA) уступают статистическим на данном датасете по причине малого объёма: 27 дней обучения недостаточны для надёжного обобщения градиентным бустингом. При расширении датасета до 3-6 месяцев и добавлении внешних регрессоров (GHI, температура, облачность) ожидается инверсия: ML-методы выйдут на MAE < 1 000 кВт.

**Нейросетевые модели (NHITS, NBEATS, LSTM)** на 34-дневном датасете не показывают преимущества перед классическими методами. Их применение оправдано при датасете от 6 месяцев и выше.
